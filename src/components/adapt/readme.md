# 跨端适配组件（`src/components/adapt`）

这层组件的目标是：**让业务代码尽量只依赖一套稳定的“跨端接口”**，从而在 H5 / 小程序 / RN 之间迁移时，把改动收敛到“实现层”，而不是全量改业务页面。

当前项目里：

- **H5 实现**：用 `@arco-design/mobile-react`（例如 `Button`、`Input`）
- **基础抽象**：用 DOM/React 直接实现（例如 `View`、`Text`、`ScrollView`、`Image`）

---

## 什么时候应该用它

- **想要跨端复用页面/组件**：优先使用 `adapt` 组件（而不是直接写 `div/span` 或直接引入 UI 库组件）。
- **业务组件准备沉淀为“可跨端模块”**：把外部依赖（UI 库、DOM 事件细节）隔离在这一层。

---

## 代价与风险（请务必先接受再使用）

使用 `adapt` **必然会带来“开发不适”**，这是用跨端能力换来的成本，常见表现：

- **功能上会受限**：`adapt` 倾向于只暴露“多端最小公约数”的能力，H5 上 UI 库（Arco）的高级 props/特性往往不能直接用。
- **开发体验会下降**：你不能随手 import Arco 组件“想怎么用就怎么用”，需要等 `adapt` 提供接口或补齐映射。
- **调试链路变长**：`业务 → adapt → Arco/DOM` 多一层封装，排查问题需要在两层之间来回看。
- **维护成本增加**：适配层需要持续维护（props 语义、版本升级、各端差异、降级策略），否则会逐渐“名存实亡”。
- **一致性难以完美**：同一 API 在不同端的表现很难 100% 一致（滚动、输入法、手势、字体渲染等）。

结论：**跨端不是免费午餐**。如果你追求的是“H5 极致体验/极快迭代”，强行全量适配反而会拖慢。

---

## 推荐策略（降低不适感）

- **分层使用**：
  - **沉淀业务模块**（需要跨端复用）→ 强制用 `adapt`
  - **一次性页面/强交互页面**（只服务 H5）→ 允许直接用 Arco（但要接受未来迁移成本）
- **先适配高频基础件**：容器/文本/按钮/输入/弹层/列表/导航优先，复杂表单、富交互先别硬塞进 `adapt`。
- **接口优先稳定**：宁愿少暴露一些 props，也不要频繁改动接口（否则业务侧会被反复打断）。

---

## 重要约束（Next.js Server/Client 边界）

Next.js App Router 中，**Server Component 不能携带事件处理器**（如 `onClick/onChange`）并传给 Client Component。

因此本目录做了两条约束：

- **默认安全**：`View/Text/Image` 只有在你显式传入 `onClick/onTap` 时才会挂事件，避免 SSR/预渲染时报错。
- **需要交互时**：把使用交互 props 的那段组件/页面写成 `"use client"`（推荐“局部 client”，不要整站都 client）。

---

## 组件清单（会逐步扩展）

- **布局/容器**：`View`
- **文本**：`Text`
- **按钮（Arco）**：`Button`
- **输入框（Arco）**：`Input`
- **图片**：`Image`
- **滚动容器**：`ScrollView`

统一从这里引入：

```ts
import { View, Text, Button, Input, Image, ScrollView } from "@/components/adapt";
```

---

## 设计原则

- **面向 Taro 的语义**：props 命名尽量贴近 Taro（例如 `onTap`、`onInput`、`scrollY` 等），让未来迁移到小程序/RN 更平滑。
- **实现可替换**：业务侧不直接依赖 `@arco-design/mobile-react`；由 `adapt` 内部决定 H5 具体用哪个库实现。
- **先小后大**：优先覆盖高频基础组件（容器/文本/按钮/输入/弹层/列表/导航），再补齐复杂组件（表单、选择器等）。

---

## 后续多端实现建议（规划）

当需要适配小程序/RN 时，建议把实现拆成多份文件或目录，例如：

- `Button.web.tsx`（H5/Arco）
- `Button.rn.tsx`（RN）
- `Button.weapp.tsx`（小程序）

业务侧 import 路径不变，靠 bundler/平台入口选择对应实现。
